# Structure du Client

[← Retour à l'index](README.md)

`src/SeriousGame.Client` est une application console/TUI qui pilote le flux de lobby décrit dans [architecture.md](architecture.md). Trois responsabilités, trois emplacements : la présentation et la saisie dans `App`, le transport hub dans `LobbyServices`, l'état dans `ClientSession`.

- `App.cs` possède la navigation **et toute l'E/S console** : prompts (pseudo, nom de partie, sélection dans la liste), rendu des écrans (menu, liste des parties, salle d'attente, animation), détection clavier. Le menu se répète jusqu'à Quitter — chaque issue ramène au menu. Les flux create/join collectent la saisie puis appellent `CreateGameAsync(gameName)` / `JoinGameAsync(gameId)` et routent sur l'`EnrollmentResult` retourné (`GameStarting` / `WaitingForPlayers` / `Failed`) ; les cas "aucune partie" et "retour au menu" sont décidés dans `App` avant tout appel hub. `App` s'abonne aux événements du service (`WaitingForPlayers`, `GameStarting`, `NotificationReceived`, `ConnectionLost`, `ConnectionRestored`) pour l'affichage — les broadcasts de salle d'attente ne sont rendus que pendant la phase d'attente (garde booléenne), pour ne pas écraser le menu pendant la navigation. Pendant l'attente, Échap annule via un `CancellationTokenSource` (sondage `Console.KeyAvailable` 10 fois/s, désactivé quand stdin est redirigé) ; sur annulation, `App` appelle `LeaveGameAsync` (méthode hub `LeaveGame` : retrait de la partie et du groupe SignalR côté serveur, sans se déconnecter) avant de revenir au menu.
- `State/ClientSession.cs` (injectée en Singleton) détient l'état : `PlayerId` (identité persistée — au démarrage, la session réutilise le premier fichier `client_id_*.txt` présent et n'en crée un que s'il n'en existe aucun, l'identifiant est donc stable d'une exécution à l'autre), `Nickname`, `Games`, `CurrentGame`. Ces champs sont en lecture seule de l'extérieur ; ils ne changent que via des méthodes nommées (`Identify`, `UpdateGames`, `EnterGame`, `LeaveCurrentGame`, `MarkPlayerInactiveAfterOtherQuit`) — un passage dans le lobby ou une partie se lit comme un appel explicite (`EnterGame`/`LeaveCurrentGame`), pas comme une durée de vie DI.
- `Services/LobbyServices.cs` implémente `ILobbyServices` ; détient la `HubConnection` vers `/lobby`, relaie les broadcasts `ILobbyHubClient` en événements .NET et met à jour la `ClientSession` via ses méthodes de mutation — **aucune E/S console** dans ce service. Les erreurs attrapées (échec de connexion, `HubException`, opération invalide) partent dans `ILogger<LobbyServices>`. L'attente du démarrage est événementielle : un `TaskCompletionSource` créé avant chaque invoke d'enrôlement (pour ne pas rater un `GameStarting` arrivé tôt sur un thread SignalR) est résolu par le handler `GameStarting` ; `WaitForGameStartAsync(CancellationToken)` l'attend via `Task.WaitAsync` et retourne `false` si le token est annulé.
- `Services/Interfaces/` regroupe les interfaces de service (`ILobbyServices`, `ILobbyService`, `IGameServices`) séparément de leurs implémentations. `ILobbyServices` (le vrai service utilisé) est distinct de `ILobbyService` (interface du prototype `RoomLobbyPrototype.cs`, inutilisé) — noms proches mais sans rapport.
- `UI/ConsoleUI.cs` / `ConsoleAnimator.cs` sont les seuls helpers d'E/S ; toute sortie destinée à l'utilisateur doit passer par eux plutôt que par du `Console.Write` brut.

## Injection de dépendances

`Program.cs` est l'unique composition root : un `ServiceCollection` enregistre `IOptions<WebSocketServerOptions>`, le logging (`AddLogging` avec le provider console), et les services de l'app (`ClientSession`, `ILobbyServices` → `LobbyServices`, `IGameServices` → `GameServices`, `App`), tous en `Singleton` — une seule instance de chacun pour toute l'exécution, résolue directement depuis le `ServiceProvider` racine (pas de scope). Contrairement au Server (où SignalR crée un scope DI par appel de hub, d'où le `Scoped` de `GameService`/`PlayerService`/`LobbyFlowService`), le Client tourne dans un seul process sans notion d'appel concurrent à isoler : `App` représente tout le run, `ClientSession`/`LobbyServices` doivent survivre à chaque passage dans le lobby (la connexion hub ne doit pas être recréée, la liste `Games` doit rester à jour même entre deux parties) — un scope par passage aurait soit reconnecté le hub à chaque retour au menu, soit vidé `Games` sans garantie de broadcast pour la repeupler.

## Config et textes UI

- **Valeurs de config** (scheme/domaine/port du hub) vivent dans `appsettings.json` sous `WebSocketServer`, injectées via le pattern `IOptions<T>` dans `LobbyServices`, qui construit l'URL du hub à partir de `options.Value` + `HubRoutes.Lobby` — aucune URL codée en dur.
- **Textes UI et ressources** (prompts de menu, en-têtes, littéraux techniques comme le nom de fichier client-id) : la convention resx/Constants (fichiers `.resx`/`Designer.cs`, quels littéraux vont où) est décrite dans `CLAUDE.md`, pour éviter de la dupliquer à deux endroits.
